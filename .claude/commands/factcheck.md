# ファクトチェック・スキル（ニンジャクラン方式）

技術記事のファクトチェックを3フェーズ・ニンジャクラン方式で実施するスキル。
記事中の技術的事実（バージョン番号、コマンド、仕様、URL、固有名詞等）を検証し、誤りがあれば修正する。

## 入力

- 記事ファイルパス: `$ARGUMENTS`（例: `articles/raspberry-pi-pxe-iscsi-boot.md`）
- パスが未指定の場合は `AskUserQuestion` で記事ファイルを確認する

## 全体フロー

```
フェーズ1: 偵察（シャドウアイ） ─ 記事精査・チェック対象抽出
    ↓
フェーズ2: 並列検証（ヴェリファイア部隊） ─ Web 検索・公式ドキュメント照合
    ↓
フェーズ3: 修正（インクブレード） ─ 記事修正・textlint チェック
    ↓
完了: ユーザーレビュー → イクサの記録保存 → クラン解散
```

---

## フェーズ1: 偵察（シャドウアイ）

グランドマスター（自身）が記事を精査し、ファクトチェック対象を洗い出す。

### 手順

1. `Read` で記事全文を読み込む
2. 以下の観点でファクトチェック対象を抽出する:
   - **バージョン番号・リリース日**: ソフトウェアのバージョン、リリース年月
   - **コマンド・設定値**: 実行コマンド、設定ファイルのパラメータ名・値
   - **技術仕様**: プロトコル仕様、API 仕様、ハードウェアスペック
   - **URL・リンク**: 参照先 URL の有効性
   - **固有名詞**: プロジェクト名、組織名、ツール名の表記
   - **因果関係・論理**: 技術的な因果関係や説明の正確性
3. 抽出結果を `plans/<date>-factcheck-suspects.md` に書き出す

### suspects ファイルのフォーマット

```markdown
# ファクトチェック対象リスト

対象記事: `<記事パス>`
作成日: YYYY-MM-DD
抽出件数: N 件

## チェック対象

### 1. <対象の簡潔な説明>
- **記事中の記述**: 「（該当箇所の引用）」
- **カテゴリ**: バージョン / コマンド / 仕様 / URL / 固有名詞 / 論理
- **検証方針**: （どこで確認すべきか。公式ドキュメント URL、man page 等）
- **優先度**: 高 / 中 / 低

### 2. ...
```

---

## フェーズ2: 並列検証（ヴェリファイア部隊）

### クラン結成

1. `TeamCreate` でファクトチェック・クランを結成する（チーム名: `factcheck`）
2. suspects の件数に応じてヴェリファイア部隊の人数を決定:
   - **6件以下**: 2人（alpha, bravo）
   - **7件以上**: 3人（alpha, bravo, charlie）
3. suspects を均等に分割し、各ニンジャに割り当てる
4. `TaskCreate` で各ニンジャのタスクを作成する

### ニンジャ spawn 時のプロンプト必須事項

各ヴェリファイアを `Agent` (subagent_type: `general-purpose`) で spawn する際、以下を必ずプロンプトに含める:

```
あなたはファクトチェック・クランのヴェリファイア「<ニンジャネーム>」である。
グランドマスターの命により、技術記事のファクトチェックをジッコウする。

【オキテ】
- 完了報告は SendMessage で1回のみ。反復報告は不要。送信後はザゼンして沈黙を保て。
- ファイル編集は行わない。結果ファイルへの Write のみ許可。記事本体には触れるな。
- 検証結果は plans/<date>-factcheck-results-<name>.md に書き出せ。

【担当チェック項目】
（suspects から割り当てた項目をここに列挙）

【検証手順】
1. WebSearch または WebFetch で公式ドキュメント・信頼できるソースを確認
2. 各項目について「正確 / 不正確 / 要確認」の判定を下す
3. 不正確な場合は正しい情報とソース URL を記載する
4. 結果ファイルを Write で保存する
5. グランドマスターに SendMessage で完了報告（1回のみ）
```

### results ファイルのフォーマット

```markdown
# ファクトチェック結果: <ニンジャネーム>

対象記事: `<記事パス>`
検証日: YYYY-MM-DD
担当: <ニンジャネーム>

## 検証結果

### 1. <対象の説明>
- **判定**: 正確 / 不正確 / 要確認
- **記事中の記述**: 「（引用）」
- **検証結果**: （調査で判明した事実）
- **ソース**: [タイトル](URL)
- **修正案**: （不正確の場合のみ。正確な記述の提案）

### 2. ...

## サマリー
- 正確: N 件
- 不正確: N 件
- 要確認: N 件
```

### 完了待ち

全ヴェリファイアの完了報告を待つ。全員完了後にフェーズ3へ進む。

---

## フェーズ3: 修正（インクブレード）

グランドマスター（自身）が検証結果を集約し、記事を修正する。

### 手順

1. 全 results ファイルを `Read` で読み込む
2. 「不正確」と判定された項目を集約する
3. 「要確認」の項目についてはユーザーに `AskUserQuestion` で判断を仰ぐ
4. 記事を `Edit` で修正する:
   - 不正確な記述を正しい情報に置換
   - 必要に応じてソース URL を脚注やリンクとして追加
5. `Bash` で textlint を実行して校正チェック:
   ```
   pnpm exec textlint <記事パス>
   ```
6. textlint エラーがあれば修正する

---

## 完了手順

### 1. ユーザーレビュー

- `git diff` で変更差分を表示し、ユーザーに確認を求める
- ユーザーの承認を待つ（クラン解散はユーザー承認後）

### 2. イクサの記録保存

ユーザー承認後、`plans/<date>-factcheck-clan-log.md` に以下を書き出す:

```markdown
# ファクトチェック・クラン イクサの記録

対象記事: `<記事パス>`
実施日: YYYY-MM-DD

## クラン編成
- グランドマスター: メインエージェント（偵察・修正担当）
- <ニンジャネーム alpha>: ヴェリファイア（検証項目 1-N 担当）
- <ニンジャネーム bravo>: ヴェリファイア（検証項目 N-M 担当）
- <ニンジャネーム charlie>: ヴェリファイア（検証項目 M-L 担当）※3人編成時のみ

## 各隊員の行動サマリー

### <ニンジャネーム>
- **担当タスク**: （担当した検証項目の概要）
- **主要な調査結果**: （発見した事実・誤りの概要）
- **発生した問題と対処**: （あれば記載）

## 検証結果サマリー
- チェック対象総数: N 件
- 正確: N 件
- 不正確（修正済み）: N 件
- 要確認（ユーザー判断）: N 件

## 修正内容
（記事に加えた修正の一覧）
```

### 3. クラン解散

- 全ニンジャに `SendMessage` (type: `shutdown_request`) を送信
- `TeamDelete` でクランを解散する
